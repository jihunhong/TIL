# Controlled vs Uncontrolled Component

React에서 폼·입력 요소를 다룰 때, **값의 단일 진실 공급원(source of truth)**이 React state인지 DOM인지에 따라 나뉜다.

## 한 줄 요약

| 구분 | 진실 공급원 | 특징 |
|------|-------------|------|
| **Controlled** | React state(또는 props로 내려온 값) | 화면에 보이는 값은 항상 state를 따른다. `onChange`로 state를 바꿔야 UI가 갱신된다. |
| **Uncontrolled** | DOM(`input` 자체) | 초기값만 `defaultValue` 등으로 주거나, 필요할 때 **ref**로 읽는다. |

## 코드에서의 구분

- **Controlled**: `value={...}` + `onChange`(또는 동등한 핸들러)로 state와 동기화.
- **Uncontrolled**: `defaultValue` 또는 비제어 + **ref**로 `inputRef.current.value` 조회. 전형적으로 **`value` 없이** 쓴다.

> 주의: `value`만 두고 `onChange`가 없으면(읽기 전용이 아닌 경우) 사용자 입력이 막히는 **반쯤 controlled** 버그가 자주 난다.

## 적용이 어렵게 느껴지는 이유

1. **폼 전체를 controlled로 묶을 때**  
   필드가 많거나 중첩 객체·배열을 `value`로 들고 있으면 리렌더, 검증 타이밍, 연동이 복잡해진다.

2. **외부에서 값이 바뀌는 경우**  
   서버 프리필, 리셋, 부모 props 갱신 등. **누가 최종 진실인지**를 명확히 해야 한다.

3. **특수 입력**  
   `<input type="file">` 등은 문자열 state로 완전히 들고 다니기 어렵거나 패턴이 달라 **ref·비제어**가 자연스럽다.

4. **라이브러리·마스킹·에디터**  
   DOM과의 동기화가 까다로워 완전 controlled가 힘들거나, **혼합**(핵심만 state, 나머지 ref)이 나온다.

## 선택 기준 (실무)

| 상황 | 보통 유리한 쪽 |
|------|----------------|
| 즉시 검증, 포맷팅, 조건부 UI, 다른 컴포넌트와 **같은 값 공유** | **Controlled** |
| 단순 제출용, 필드 많고 **제출 시점에만** 값이 필요 | **Uncontrolled + ref** 또는 폼 라이브러리(react-hook-form 등) |
| 입력이 커서 리렌더가 부담 | 전부 state보다 **제한된 state** 또는 라이브러리 검토 |

## 혼합 패턴도 현실적이다

- 대부분 uncontrolled + 제출 시 수집, **특정 필드만** controlled로 실시간 검증·연동.
- 대부분 controlled인데 **파일만 ref** 등.

## React Hook Form과의 관계

`react-hook-form`(RHF)은 위 논의와 직접 연결된다. 요지는 **기본은 비제어에 가깝게 두고, 필요할 때만 제어 쪽으로 올린다**는 것이다.

### RHF가 보통 하는 일

- **`register`**로 `name`, `ref`, 검증 규칙을 input에 연결한다. 값은 DOM이 들고 있고, 제출·검증 시점에 RHF가 ref로 읽는다 → **전통적인 의미의 uncontrolled에 가깝다.**
- 그래서 **모든 필드를 `useState`로 두는 controlled 폼**보다 **리렌더가 적은 편**인 경우가 많다.

### 완전 비제어만은 아님

- **`watch` / `useWatch`**: 구독하는 필드가 바뀔 때 컴포넌트가 다시 그려질 수 있어, 그 부분은 **React가 값 변화를 알아야 하는** 패턴에 가깝다.
- **`setValue`**: 외부에서 값을 넣어 **DOM과 RHF 내부 상태를 같이 맞춘다.** 프리필·리셋·연동에 쓴다.
- **`Controller` / `useController`**: MUI `TextField`, 날짜 피커 등 **내부적으로 `value`/`onChange`를 요구하는 컴포넌트**에 쓰면, 그 필드는 **controlled에 가깝게** 다룬다.

### RHF API와 controlled/uncontrolled

| RHF API | controlled/uncontrolled와의 관계 |
|---------|----------------------------------|
| `register` | 기본 입력은 **비제어 스타일** (DOM + ref) |
| `Controller` | 서드파티·복잡한 입력은 **제어 컴포넌트로 감싸서** 연동 |
| `watch` / `useWatch` | 실시간으로 UI가 따라가야 할 때 — **구독 범위**만 불필요하게 넓히지 않기 |
| `setValue` / `reset` | 서버 값·리셋 등 **진실을 한 번에 맞출 때** |

### 실무에서 한 줄로

- 단순 `<input>`: `register`로 두고, 실시간으로 다른 곳에 반영해야 할 때만 `watch`/`useWatch`를 **좁게** 쓴다.
- 라이브러리 컴포넌트: `Controller`로 **value/onChange를 RHF에 연결**한다 (이때는 controlled 논의가 그대로 적용된다).

## 이해 체크 질문

1. 이 입력의 **하나의 진실**은 React state인가, DOM인가?
2. **매 입력마다** 부모·형제가 그 값을 알아야 하는가?
3. **외부에서** 값을 강제로 바꿔야 하는가(리셋, 프리필 등)?

## Controller / useController

### 하는 일

- **`Controller`** — RHF 가 `ref` 만으로는 못 붙이는 컴포넌트(MUI `TextField`, React Select, 날짜 피커 등)에 쓴다. **자식에게 `value`(또는 `checked`)랑 `onChange` 를 넣어 줘야** 하는 타입이다.
- **`render` prop** — RHF 가 넘겨 주는 `field` (`value`, `onChange`, `onBlur`, `name`, `ref`) 를 그대로 자식에 연결한다. 즉 **그 필드만큼은 controlled 연동**이라고 보면 된다.
- **`useController`** — `Controller` 랑 같은 역할인데 훅 형태다. 컴포넌트를 한 겹 더 안 감싸도 되서 **스타일/구조 맞출 때** 편한 경우가 있다.

### 왜 굳이 쓰나
- `register` 는 네이티브 `<input>` 처럼 **ref + name** 으로 끝나는 쪽에 잘 맞는다.
- 그런데 써드파티는 내부에서 state 를 쓰거나 API 가 `value`/`onChange` 고정이라 **`register` 만으로는 연결이 안 되는 경우**가 많다. 그때 `Controller` 로 **필드 값을 RHF 상태와 동기화**한다.

### 쓸 때 헷갈리는 점
- **이름(`name`)** 은 `Controller` 의 `name` 과 자식에 넘기는 `field.name` 이 맞아야 한다.
- **`defaultValue`** — controlled 필드의 초기값은 보통 `defaultValues`(폼 단위)나 `Controller` 의 옵션으로 맞춘다. 자식 컴포넌트마다 규칙이 다르면 문서를 한 번 본다.
- **성능** — `watch` 처럼 구독 범위가 넓어지지 않게, **필요한 필드만** `Controller` 로 감싸는 게 좋다는 말이 나오는 이유와 같다 (리렌더가 늘 수 있음).

## formState 는 왜 구조 분해해서 쓰나

`useForm()` 이 돌려주는 `formState` 는 그냥 객체가 아니라 **Proxy** 로 감싸져 있다. 그래서 **구독(subscription)** 이 “전체 `formState` 한 덩어리” 가 아니라 **렌더에서 실제로 읽은 속성 단위**로 열린다.

### Proxy 가 뭔가

다른 객체(`target`)를 감싸고, 그 객체에 대한 **읽기·쓰기를 중간에서 가로챌 수 있게** 하는 빌트인이다. `new Proxy(target, handler)` 에서 `handler` 안의 `get`, `set` 등을 **트랩(trap)** 이라고 부른다.

```js
const target = { count: 0 };

const handler = {
  get(obj, prop) {
    console.log("읽음:", prop);
    return obj[prop];
  },
};

const p = new Proxy(target, handler);
p.count; // "읽음: count" 로그 후 0
```

- **왜 쓰나** — 접근 로깅·디버깅, `set` 으로 쓰기 검증, 실제로는 없는 키인데 `get` 에서 계산해 돌려주기, 지연 계산 등.
- **RHF `formState` 와** — 대략 **`get` 트랩에서 “이번 렌더에서 이 키를 읽었다”고 잡아 두고**, 그 속성만 구독에 건다고 보면 된다 (내부 구현·버전마다 디테일은 다를 수 있음).

| 트랩 | 언제 |
|------|------|
| `get` | `obj.prop` 읽을 때 |
| `set` | `obj.prop = value` 할 때 |
| `has` | `prop in obj` |
| `deleteProperty` | `delete obj.prop` |

한 줄로는 **객체 접근을 가로채는 래퍼** 정도로 기억해두면 된다.

- **리렌더를 필요한 조각에만 걸기 위해서** — `errors`, `isDirty`, `isSubmitting` 등은 바뀌는 타이밍이 다르다. RHF 는 **그때그때 읽힌 키**만 구독에 걸어서, 그 값이 바뀔 때만 다시 그리게 해둔다.
- **구독은 “그 속성을 읽는 순간”에 붙는다** — `formState.errors` 처럼 **특정 키에 접근할 때** 구독이 잡힌다. 그래서 문서에서도 렌더 안에서 필요한 필드를 **미리 읽어라**(보통 `const { errors, isDirty } = formState`)고 한다.  
  `formState` 통째로만 넘기거나, 첫 접근을 `useEffect` 안에만 두면 **구독이 기대대로 안 잡거나** 업데이트를 놓칠 수 있다.
- **성능** — `formState` 안을 전부 구독하면 입력할 때마다 불필요한 리렌더가 늘 수 있어서, **쓰는 필드만** 골라 구독하게 만든 설계에 가깝다.

정리하면, **`formState` 는 “전체 구독”이 아니라 “읽은 속성만 구독”이라서, 쓸 값은 구조 분해로 꺼내 쓰는 패턴이 권장된다**고 보면 된다.

### 실무에서

- `const { errors, isSubmitting, isDirty } = formState` 처럼 **필요한 것만** 꺼낸다.
- 자식에 `formState` 전체를 넘기기보다 **필요한 필드만** 넘기거나, 자식에서 렌더 시점에 다시 꺼내 읽게 하는 편이 낫다.

## `watch` 와 `useWatch`

둘 다 RHF에서 **폼 입력값을 실시간으로 보고 싶을 때** 쓰지만, **리렌더가 퍼지는 범위**에서 차이가 크다. 성능을 볼 때는 이 관점이 핵심이다.

### 1. `watch` (`useForm` 이 돌려주는 함수)

`useForm()`에서 꺼내 쓰는 함수다.

- **동작** — 감시할 필드를 지정하면 그 값 변화에 반응한다 (구독).
- **리렌더** — `watch`를 호출한 **그 컴포넌트**가 값이 바뀔 때마다 다시 그려진다. `useForm`을 쓰는 컴포넌트가 보통 폼의 **최상단**이면, 그 컴포넌트 전체가 리렌더링된다.
- **장점** — 사용이 단순하고, 폼 루트에서 조건부 UI·전체 제출 버튼 활성화처럼 **한곳에서** 흐름을 잡기 좋다.
- **주의** — 폼이 커질수록 최상단이 자주 그려지면 **하위 전체가 따라 도는** 비용이 커질 수 있다.

### 2. `useWatch` (별도 훅)

`useWatch({ control, name })` 형태로 쓰는 독립 훅이다.

- **동작** — 지정한 필드(들)만 구독한다.
- **리렌더** — 값이 바뀔 때 **`useWatch`를 호출한 컴포넌트만** 리렌더링될 수 있다. 부모(Form 루트)는 그 때문에 자동으로 다시 그려지지 않는다.
- **장점** — 큰 폼에서 **특정 구역만** 실시간 반응할 때 부모의 무거운 로직·자식 트리를 덜 흔든다.

### 3. 비교

| 항목 | `watch` | `useWatch` |
|------|---------|------------|
| **가져오는 곳** | `useForm()` 반환값 | 별도 훅 |
| **리렌더가 잘 나는 곳** | `watch`를 쓴 컴포넌트(보통 폼 루트) | `useWatch`를 쓴 컴포넌트만 |
| **성능** | 폼이 작거나 루트에서만 감시할 때 부담 적음 | 폼이 크고, 하위만 바꾸고 싶을 때 유리 |
| **쓰기 좋은 때** | 최상단에서 조건부 렌더·전역 제어 | 하위에서 글자 수, 옵션 노출 등 국소 UI |

### 4. 선택 흐름

1. **폼 루트에서** 값 하나로 전체 흐름을 잡으면 된다 (제출 버튼, 단계 전환 등) → **`watch`**로도 충분한 경우가 많다.
2. **특정 자식 안에서만** 실시간 값이 필요하다 (글자 수, 세부 옵션 등) → **`useWatch`**를 그 자식에 두어 부모 리렌더를 줄인다.

### `formState`·Proxy 와 맞물리는 부분

`useWatch`도 **구독(subscription)** 모델에 가깝다. 폼 저장소는 가볍게 두고, 구독한 컴포넌트만 값 변화에 반응해 다시 그리는 쪽으로 이해하면 `formState`에서 말한 Proxy·구독과 같은 맥락으로 묶어 볼 수 있다 (내부 구현은 버전·문서 기준으로 보면 된다).
